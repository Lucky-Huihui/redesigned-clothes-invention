**文档版本**：V1.0

## 1. 产品概述 (Product Overview)
- **核心价值**：帮助用户（尤其是有大量服饰的女性）数字化管理个人衣物。通过“分类浏览”解决“找不到衣服”的痛点，通过“AI虚拟试穿”解决“出门前搭配费时”的痛点。
- **用户画像**：时尚敏感度高的年轻女性/男性，注重穿搭效率。
- **技术基调**：移动端App（适配iOS/Android），界面简洁清晰，支持双主题切换。

## 4. 功能模块详细设计

### 4.1 登录/注册模块 (Auth) —— *关键字5*
- **注册页**：提供“手机号注册”与“邮箱注册”Tab切换。
    - 输入框：手机号/邮箱、密码、确认密码。
    - 点击注册后，系统自动创建用户并初始化默认分类。
- **登录页**：输入手机号/邮箱 + 密码，点击登录进入主页。
- **安全逻辑**：密码需加密存储（如bcrypt）。

### 4.3 物品管理模块 (Items) —— *关键字2、3、7、8*
- **界面逻辑**：
    1. **主视图（分类卡片流）**：展示当前用户所有的**分类卡片**（默认+自定义）。卡片上显示分类名称和该分类下的物品数量。
    2. **分类管理**：右上角提供“管理分类”入口。点击进入新页面：
        - 显示所有分类列表。
        - 底部有“新建分类”输入框+按钮（仅非默认分类右侧显示“删除”红叉）。
    3. **分类详情页**（点击分类卡片进入）：
        - 顶部导航栏显示分类名 + “新建物品(+)”按钮。
        - 中间以**网格列表(Grid)**展示该分类下的所有物品卡片。
        - **物品卡片**展示：缩略图、物品名称、创建时间（小字）、价格（若有）。
        - **长按/左滑物品卡片**：弹出操作菜单（编辑名称、修改照片、删除）。
    4. **新建物品弹窗**：包含“上传照片（必填）”按钮、“输入名称（必填）”输入框、“输入价格（选填）”输入框，点击确认保存。

---

### 4.4 搭配试穿模块 (Outfits) —— *关键字4、9、10、11*
- **步骤一：选择分类与物品（单选约束）**
    - 界面显示所有分类（或仅显示包含物品的分类）。
    - **交互逻辑**：
        - 用户点击一个分类（如“上装”） -> 底部弹出/跳转子页面，展示该分类下的物品列表（网格）。
        - **关键约束**：物品选择为**单选**（点击选中，再次点击取消选中/替换选中）。系统需标记当前选中的分类与物品。
        - 界面顶部展示“已选清单”：如 `上装：白T恤 ✓`，`下装：未选`。
    - **必选分类校验**：系统检测是否 **“上装”**、**“下装”**、**“鞋子”** 三个分类均已选中对应物品。
        - 若未选满，**“开始试穿”按钮置灰不可点**，并提示“请至少选择上装、下装和鞋子”。
        - 若已选满，按钮高亮可点。

- **步骤二：AI试穿生成（核心）**
    - 点击“开始试穿”，系统将 **仅提取所选物品的图片URL数组** 以及 **当前用户的性别（Gender）** 发送给后端AI服务。
    - 前端展示“生成中”Loading动画。
    - 生成成功后，在界面中央以**大图卡片**形式展示 AI模特穿着所选服装的效果图。
    - **生成结果反馈区**（位于效果图下方）：三个独立按钮 —— ❤️ 喜欢、⭐ 收藏、👎 不喜欢。
        - 点击即记录反应到 `Reactions` 表，并给予Toast轻提示（“已标记喜欢”）。

### 4.5 个人中心模块 (Profile) —— *关键字6、12*
- **顶部**：显示用户头像（圆形）、昵称。
- **中间功能区**（查看互动记录）：

        - **性别切换**（男女切换）—— **联动影响**：修改后，下次生成AI试穿将使用新性别的模特。
    - **注销账号**（红色警示按钮）：点击弹出二次确认窗，确认后物理删除该用户所有数据（级联删除物品、搭配、分类）。

## 5. 核心业务逻辑规则 (Business Logic Constraints)
*（供AI编写严谨代码的校验规则）*

1.  **分类唯一性**：同一个用户下，分类名称不可重名。
2.  **物品归属唯一性**：创建物品时，必须且只能选择一个分类。**严禁**一个物品同时出现在两个分类下（外键约束保证）。
3.  **搭配互斥性**：在搭配选择流程中，当用户在一个分类下选中一个物品后，若再点击同分类的其他物品，新物品**替换**旧物品，不可多选（对应关键字4）。
4.  **AI生成触发条件**：后端接口必须校验请求参数中是否包含 `上装`、`下装`、`鞋子` 三个分类的Item ID，缺一不可（对应关键字9）。
5.  **反应覆盖逻辑**：如果用户对搭配A点了“喜欢”，再点“收藏”，则原先的“喜欢”记录被删除，仅保留最新的“收藏”（或根据UI设计变为同时存在，但建议同用户同搭配仅存一条反应，类型更新）。

## 7. 接口/API 伪代码逻辑建议 (供后端AI参考)
- `POST /api/auth/register` : { loginAccount, password } -> 自动创建默认分类。
- `GET /api/categories` : 返回该用户所有分类（包含是否为默认）。
- `POST /api/items` : { categoryId, name, price, imageBase64 } -> 存储。
- `POST /api/outfits/generate` : { itemIds: [id1, id2, id3], gender } -> 调用外部AI绘图服务，返回生成的ImageUrl。
- `POST /api/reactions` : { outfitId, type } -> 覆盖写入反应表。
